## 7/15 (환경 세팅 & 프로젝트 초기화)

### 1. 터미널에 (base)가 계속 표시됨
- **증상**: 터미널을 열 때마다 프롬프트 앞에 `(base)`가 붙음
- **원인**: 예전에 설치한 Anaconda가 자동으로 켜짐. pyenv와 파이썬 관리 주체가 겹침
- **해결**: `conda config --set auto_activate_base false`
- **배운 점**: 파이썬 버전 관리 도구를 두 개 같이 쓰면 어떤 파이썬이 실행될지 알 수 없음. 한 프로젝트에는 하나만

### 2. python 명령어를 못 찾음
- **증상**: `python -m venv .venv` 실행 시 `command not found: python`
- **원인**: 맥에는 `python`이 없고 `python3`만 있음
- **해결**: `python3 -m venv .venv`로 실행
- **배운 점**: 맥·리눅스에서는 `python3`이 기본. 가상환경을 켠 뒤에는 `python`도 씀

### 3. pyenv로 버전을 바꿨는데 적용이 안 됨
- **증상**: `pyenv version`은 3.12.13인데 `python3 --version`은 3.9.6
- **원인**: `.zshrc`에 pyenv 시작 설정이 없어서, pyenv를 거치지 않고 맥에 원래 있던 파이썬이 실행됨
- **해결**: `.zshrc`에 pyenv 초기화 3줄 추가 후 `source ~/.zshrc`
- **배운 점**: pyenv는 파이썬 호출을 중간에서 가로채는 방식. 셸 설정에 등록해야 작동함

### 4. 브랜치 이름이 master로 만들어짐
- **증상**: GitHub은 `main`인데 내 컴퓨터만 `master`
- **원인**: 저장소를 만들 때 옛날 기본값이 적용됨
- **해결**: `git branch -m master main` (이름 바꾸기)
- **배운 점**: `init.defaultBranch main` 설정은 앞으로 만들 저장소에만 적용됨. 이미 만든 건 직접 바꿔야 함

### 5. PR이 만들어지지 않음
- **증상**: `gh pr create` 실행 시 "두 브랜치가 같다"는 에러
- **원인**: 작업 브랜치를 먼저 올려서 그게 기본 브랜치가 됨. 그 뒤 만든 main과 내용이 완전히 같았음
- **해결**: main을 올리고 `gh repo edit --default-branch main`으로 기본 브랜치 변경
- **배운 점**: main을 먼저 올리고 작업 브랜치를 나누는 순서가 안전함. PR은 두 브랜치에 차이가 있어야 만들 수 있음

## 7/16 (회원가입 API + MySQL 연동)

### 1. 짧은 비밀번호인데 "72바이트 초과" 에러
- **증상**: 비밀번호가 `1234`인데도 72바이트를 넘는다는 에러가 남
- **원인**: 비밀번호 길이 문제가 아니라, passlib과 최신 bcrypt(4.1 이상)가 서로 맞지 않아서 생긴 오류
- **해결**: `pip install 'bcrypt<4.1'`로 버전 고정, requirements.txt에 기록
- **배운 점**: 에러 메시지가 알려주는 원인과 실제 원인이 다를 수 있음. 라이브러리 충돌은 버전을 고정하는 게 표준 해법

### 2. 같은 계정인데 터미널에서만 접속이 거부됨
- **증상**: 백엔드는 MySQL에 잘 붙는데, `docker exec`로 같은 계정 접속하면 거부됨
- **원인**: MySQL은 `localhost`(소켓)와 `127.0.0.1`(네트워크)을 다른 접속으로 봄. appuser에는 localhost 권한이 없었음
- **해결**: 확인이 목적이었으므로 root로 조회. appuser 권한은 넓히지 않기로 결정
- **배운 점**: 앱이 실제로 쓰지 않는 경로를 위해 권한을 늘리는 건 최소 권한 원칙에 어긋남. 고치지 않는 것도 판단임

## 7/20 (옷장 CRUD 구현)

### 1. 에러는 없는데 상세 조회만 안 됨
- **증상**: 목록 조회는 되는데 `GET /garments/1`만 계속 404. 코드에 에러 표시 없음
- **원인**: 조회 조건을 `Garment.user_id == garment_id`로 씀. 앞쪽은 `Garment.id`여야 함. 결과적으로 절대 성립할 수 없는 조건이 됨
- **해결**: `.filter(Garment.id == garment_id, Garment.user_id == user_id)`
- **배운 점**: 문법이 맞으면 에러가 안 나서 편집기도 못 잡음. 실제로 요청을 보내봐야 발견됨. 자동으로 잡으려면 테스트 코드 필요

### 2. Swagger에 토큰 넣을 칸이 없음
- **증상**: Authorize를 눌렀는데 토큰 칸 대신 아이디·비밀번호 칸만 나옴
- **원인**: `OAuth2PasswordBearer`를 씀. 이 방식은 Swagger가 form 형식으로 로그인하는데, 이 프로젝트 로그인 API는 JSON을 받음
- **해결**: `HTTPBearer`로 교체. 토큰만 확인하는 방식이라 입력칸이 하나만 나옴
- **배운 점**: 같은 JWT라도 방식에 따라 Swagger가 요청을 만드는 형태가 달라짐. 로그인 API 형식에 맞춰 골라야 함

## 7/21

### 1. 컬럼을 추가했는데 테이블에 반영되지 않음
- **증상**: 모델에 컬럼을 넣고 테이블을 지운 뒤 서버를 껐다 켰는데 테이블이 다시 생기지 않음
- **원인**: `main.py`에는 테이블 생성 코드가 없음. 테이블은 `init_db.py`를 직접 실행할 때만 만들어짐
- **해결**: `DROP TABLE` 후 `python init_db.py` 실행, `DESC`로 컬럼 확인
- **배운 점**: 서버 재시작과 테이블 생성은 별개. 컬럼을 추가했으면 지우고 다시 만들어야 하는데, 이 방법은 데이터가 같이 사라져서 운영 환경에선 못 씀(Alembic 필요)

### 2. 음수 검증을 넣었는데 음수가 통과함
- **증상**: `Field(gt=0)`을 추가했는데 음수가 그대로 저장됨
- **원인**: 조건을 기존 치수 필드에 붙이지 않고 `gt`라는 필드를 새로 만들어버림
- **해결**: `total_length_cm: Decimal | None = Field(default=None, gt=0)`처럼 기본값 자리를 Field로 교체
- **배운 점**: Field는 새 필드를 만드는 게 아니라 있는 필드에 조건을 붙이는 도구. 검증을 넣은 뒤엔 반드시 잘못된 값을 보내 422가 나오는지 확인해야 함

### 3. 수정 API에만 .id를 빼먹음
- **증상**: 서버는 정상인데 PUT 요청만 항상 실패하는 상태
- **원인**: 서비스에 회원번호(`current_user.id`) 대신 사용자 정보 전체(`current_user`)를 넘김. 숫자와 객체를 비교하니 아무것도 안 걸림
- **해결**: `current_user.id`로 수정
- **배운 점**: 7/20 조회 조건 오류와 같은 유형. 엔드포인트 5개를 나란히 놓고 인자가 같은 모양인지 비교하는 습관이 필요하고, 결국 테스트 코드로 자동 확인해야 함

### 4. 카테고리를 입력했는데 DB에는 NULL로 저장됨
- **증상**: 옷장 등록 시 `category`를 보냈는데 저장이 안 됨. 등록 시점엔 에러가 없었고, 나중에 비교 API에서 400(카테고리 불일치)으로 뒤늦게 드러남
- **원인**: DB 컬럼은 있는데 `GarmentCreate` 스키마에 `category` 필드가 없어서 pydantic이 그 값을 조용히 버림
- **해결**: `GarmentCreate`·`GarmentResponse`에 `category: str | None = None` 추가. 컬럼은 이미 있어서 `DROP TABLE` 불필요. 기존 데이터는 `UPDATE`로 채움
- **배운 점**: DB 컬럼 추가와 스키마 추가는 별개의 작업. 스키마에 없는 값은 에러 없이 사라지기 때문에 엉뚱한 곳에서 증상이 나타남. 응답만 보지 말고 `SELECT`로 실제 저장값을 확인하는 게 빠름

### 5. MySQL에서 한글이 ????로 보임
- **증상**: `SELECT` 결과의 한글 컬럼이 전부 `????`로 표시됨
- **원인**: 컬럼 문자셋은 `utf8mb4`로 정상. 접속한 클라이언트 쪽 문자셋 설정 문제였음
- **해결**: 접속 시 `--default-character-set=utf8mb4` 옵션을 붙여 재접속
- **배운 점**: `SHOW FULL COLUMNS`의 Collation을 먼저 보면 데이터가 깨진 건지 화면만 깨진 건지 구분됨. 데이터는 멀쩡했으므로 복구 작업이 필요 없었고, 원인 위치를 먼저 좁히는 게 중요함


## 7/22

### 1. pip freeze로 만든 requirements.txt에 테스트 도구 섞임
- **증상**: pip install -r requirements-dev.txt 실행 시 pytest가 requirements.txt (line 26)에서 설치됐다고 출력됨
- **원인**: pip freeze가 개발용 도구까지 전부 담아버림
- **해결**: requirements.txt에 pytest 관련 파일 모두 지운 뒤, dev파일에 -r 명령어 추가
- **배운 점**: 개발용과 운영용 의존성을 분리해야함. 에러가 없어도 예상한 출력이 안나오면 의심해야함

### 2. passlib DeprecationWarning
- **증상**: pytest에서 16 passed 했지만 1 warning으로 경고 뜸
- **원인**: passlib이 파이썬 3.13에서 없어질 부품 사용
- **해결**: 3.12 사용중이므로 동작 이상 없음. 또한 버전 고정 유지로 대응 완료
- **배운 점**: 에러는 지금 안돌아가는 것, 경고는 지금은 되지만 나중에 문제가 될 것

- **증상**: 요청을 보내도 JSON 로그 안나오고 에러도 없음

- **해결**: main.py에 basicConfig로 기준 낮춤
- **배운 점**: 에러 또는 경고가 없어도 예상한 출력이 안나오면 문제가 있는 것, 코드보다 설정을 의심해야함
